### Pattern: TEA (The Elm Architecture)
All changes follow: `handle_key() -> Option<Action> -> update() mutates AppState -> view() renders`
- New actions go in `action.rs`
- Key mappings go in `handle_key()` in `app.rs`
- State mutations go in `update()` in `app.rs`
- Rendering changes go in `ui/panels.rs`

### Pattern: refresh_needed() Domain Function (NEW)
```rust
// src/domain/command.rs (or new src/domain/refresh.rs)
pub struct RefreshSet {
    pub worktrees: bool,     // full worktree reload
    pub staleness: bool,     // re-check check_stale()
    pub jira_titles: bool,   // re-fetch JIRA titles for changed branches
}

pub fn refresh_needed(cmd: &CommandSpec) -> RefreshSet {
    match cmd {
        // Branch-changing ops -> full reload
        CommandSpec::GitCheckout { .. }
        | CommandSpec::GitCheckoutNew { .. }
        | CommandSpec::GitRebase { .. }
        | CommandSpec::GitResetHard
        | CommandSpec::GitResetHardFetch => RefreshSet {
            worktrees: true,
            staleness: true,
            jira_titles: true,
        },
        // Non-branch-changing git ops -> no reload
        CommandSpec::GitPull
        | CommandSpec::GitPush
        | CommandSpec::GitFetch => RefreshSet::none(),
        // Install/clean -> staleness refresh only
        CommandSpec::YarnInstall
        | CommandSpec::YarnPodInstall
        | CommandSpec::RmNodeModules
        | CommandSpec::RnCleanAndroid
        | CommandSpec::RnCleanCocoapods => RefreshSet {
            worktrees: false,
            staleness: true,
            jira_titles: false,
        },
        // Everything else -> no refresh
        _ => RefreshSet::none(),
    }
}
```

**Recommendation for RefreshSet type:** Use a simple struct with bool fields. Bitflags is overkill for 3 flags. This is the most readable and debuggable approach.

### Pattern: External Metro Detection
```rust
// src/infra/port.rs (extend existing module)
pub struct ExternalMetroInfo {
    pub pid: u32,
    pub working_dir: String,
}

/// Check if port 8081 is occupied by a non-dashboard process.
/// Uses `lsof -i :8081 -t` to get PID, then `ps -p <pid> -o comm=` for info.
pub async fn detect_external_metro() -> Option<ExternalMetroInfo> {
    // lsof -i :8081 -sTCP:LISTEN -t  -> returns PID(s)
    // ps -p <pid> -o lstart=,command=  -> returns process info
    // lsof -p <pid> +D /  -> can get cwd (slow, prefer `ps`)
    // Better: lsof -a -p <pid> -d cwd -Fn -> gets cwd field
}
```

**Recommendation for PID/cwd detection on macOS:** Use `lsof -i :8081 -sTCP:LISTEN -t` to get PID, then `lsof -a -p <PID> -d cwd -Fn` to get the working directory. The `-Fn` flag outputs field-tagged format where the `n` prefix line contains the path. This is the most reliable approach on macOS (no /proc filesystem).

### Anti-Patterns to Avoid
- **Scattered refresh logic:** Do NOT add refresh dispatches in individual `CommandExited` match arms. All refresh logic goes through `refresh_needed()`.
- **Blocking lsof in event loop:** External metro detection MUST be async (tokio::process::Command), not blocking std::process::Command.
- **Shared scroll state:** Metro and command output have separate scroll state. Do not conflate them.

## Common Pitfalls

### Pitfall 1: Staleness mtime Unreliability
**What goes wrong:** `check_stale()` compares `node_modules/` directory mtime with `package.json`/`yarn.lock` mtime. On macOS, directory mtime only updates when direct children are added/removed, NOT when files inside subdirectories change. `yarn install` creates files deep inside `node_modules/` but may not update the top-level directory mtime.
**Why it happens:** The current implementation checks `std::fs::metadata(&nm).modified()` on the `node_modules` directory itself.
**How to avoid:** Compare against a sentinel file that `yarn install` always touches. Options:
1. Check `node_modules/.yarn-integrity` mtime (yarn 1.x creates this on every install)
2. Check `node_modules/.package-lock.json` (npm creates this)
3. Check `node_modules/.cache` directory existence/mtime
**Recommendation:** Use `node_modules/.yarn-integrity` -- this file is written by yarn on every successful install and is the most reliable indicator.

### Pitfall 2: Auto-scroll vs Manual Scroll Conflict
**What goes wrong:** When user manually scrolls up to read logs, new incoming lines reset their position to bottom.
**Why it happens:** The current auto-scroll logic checks `log_scroll_offset == 0` to mean "follow tail", but 0 is also the initial state.
**How to avoid:** Use a dedicated `auto_follow: bool` flag (default true). Set to false when user scrolls up (j/k/gg). Set back to true when user presses G (go to bottom). New lines only auto-scroll when `auto_follow` is true.

### Pitfall 3: Tab Key Collision in Fullscreen
**What goes wrong:** Tab currently maps to `FocusNext` in normal mode. In fullscreen, it should exit fullscreen instead.
**Why it happens:** The fullscreen check in `handle_key()` is missing -- Tab falls through to the NORMAL MODE section.
**How to avoid:** Add fullscreen check BEFORE the normal mode Tab handling. When `state.fullscreen_panel.is_some()` and key is Tab, return `Action::ToggleFullscreen`.

### Pitfall 4: iOS Device Picker Shows Simulators
**What goes wrong:** `i>d` (iOS device picker) calls `list_ios_devices()` which runs `xcrun simctl list devices` -- this returns SIMULATORS, not physical devices.
**Why it happens:** `xcrun simctl` is the simulator management tool. Physical iOS devices are listed by different tools.
**How to avoid:** Physical iOS devices can be listed via `xcdevice list` (Xcode 15+) or by checking for `devicectl` (Xcode 16+). However, the CONTEXT.md says "fix device enumeration filtering" suggesting the current code should be FILTERING the existing output, not changing the source. Review whether the intent is: (a) show only physical devices when pressing `d`, or (b) rename/restructure to make it clear these are simulators. Given the existing `SimulatorUsed` action, the current `i>d` flow already treats these as simulators -- the bug may be that the UI labels them as "devices" when they are simulators, or that physical device support is a separate concern.

### Pitfall 5: Metro Debugger (j key) Not Working
**What goes wrong:** The `j` key in metro pane is supposed to send debugger command but may not work.
**Why it happens:** Looking at the code: `Char('J')` (uppercase/shift-J) maps to `MetroSendDebugger`, which calls `state.metro.send_stdin(b"j\n".to_vec())`. The keybinding is Shift-J, not lowercase j. The CONTEXT.md says "j key" -- need to verify if the issue is the keybinding (should it be lowercase?) or the stdin forwarding mechanism itself.

### External Metro Detection (macOS)
```rust
pub async fn detect_external_metro(port: u16) -> Option<ExternalMetroInfo> {
    // Step 1: Get PID of process listening on port
    let output = tokio::process::Command::new("lsof")
        .args(["-i", &format!(":{port}"), "-sTCP:LISTEN", "-t"])
        .output()
        .await
        .ok()?;

    let pid_str = String::from_utf8_lossy(&output.stdout).trim().to_string();
    let pid: u32 = pid_str.lines().next()?.parse().ok()?;

    // Step 2: Get working directory of that PID
    let cwd_output = tokio::process::Command::new("lsof")
        .args(["-a", "-p", &pid.to_string(), "-d", "cwd", "-Fn"])
        .output()
        .await
        .ok()?;

    let cwd_text = String::from_utf8_lossy(&cwd_output.stdout);
    let working_dir = cwd_text.lines()
        .find(|l| l.starts_with('n'))
        .map(|l| l[1..].to_string())
        .unwrap_or_else(|| "unknown".to_string());

    Some(ExternalMetroInfo { pid, working_dir })
}
```

### Staleness Fix with .yarn-integrity
```rust
pub fn check_stale(worktree_path: &Path) -> bool {
    // Primary indicator: .yarn-integrity is written on every successful `yarn install`
    let integrity = worktree_path.join("node_modules").join(".yarn-integrity");

    let integrity_mtime = match std::fs::metadata(&integrity).and_then(|m| m.modified()) {
        Ok(t) => t,
        Err(_) => return true, // no integrity file -> stale (needs yarn install)
    };

    let mut max_lock_mtime: Option<std::time::SystemTime> = None;
    for lock_file in &["package.json", "yarn.lock"] {
        let lock_path = worktree_path.join(lock_file);
        if let Ok(mtime) = std::fs::metadata(&lock_path).and_then(|m| m.modified()) {
            max_lock_mtime = Some(match max_lock_mtime {
                Some(current) => current.max(mtime),
                None => mtime,
            });
        }
    }

    match max_lock_mtime {
        Some(lock_mtime) => integrity_mtime < lock_mtime,
        None => false,
    }
}
```

## Open Questions

1. **iOS device picker intent**
   - What we know: `i>d` calls `list_ios_devices()` which uses `xcrun simctl list devices` (simulators only)
   - What's unclear: Should `i>d` show physical devices (requires different tool) or is the fix just UI labeling? The existing `SimulatorUsed` action suggests simulators are expected
   - Recommendation: Investigate during implementation. If physical device listing is needed, use `xcdevice list` or `ios-deploy --detect`. If it's a labeling issue, rename the picker

2. **Metro debugger key binding**
   - What we know: `Char('J')` (Shift-J) triggers MetroSendDebugger. The bug report says "j key" not working
   - What's unclear: Is the issue the key binding (should be lowercase j?) or the stdin pipe mechanism?
   - Recommendation: Check if `send_stdin()` actually delivers bytes to metro's stdin. The keybinding J (uppercase) is correct for avoiding collision with j=scroll-down

3. **gg two-key sequence**
   - What we know: Vim's `gg` requires two keypresses to go to top
   - What's unclear: Whether to implement true two-key sequence (with timeout) or use a simpler approach
   - Recommendation: Implement a `pending_g: bool` flag in AppState. First `g` sets the flag. Second `g` within the same event loop tick triggers scroll-to-top. Any other key clears the flag. This is simple and avoids timer complexity.
