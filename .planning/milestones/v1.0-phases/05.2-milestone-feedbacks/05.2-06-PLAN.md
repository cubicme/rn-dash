<task type="auto">
  <name>Task 1: Fix check_stale to handle multiple yarn versions</name>
  <files>src/infra/worktrees.rs</files>
  <action>
Investigate and fix check_stale() which always returns true (UAT Test 6).

The current implementation checks for `.yarn-integrity` which is a yarn v1 artifact. The UMP project may use yarn berry (v2/v3/v4) which does NOT create `.yarn-integrity`. Yarn berry uses `.pnp.cjs` or `.pnp.loader.mjs` instead (if PnP is enabled), or if using `nodeLinker: node-modules` it creates `node_modules/.yarn-state` instead.

**Fix check_stale() with a multi-sentinel approach:**

1. First, check for `node_modules/.yarn-integrity` (yarn v1 classic)
2. If not found, check for `node_modules/.yarn-state` (yarn berry with nodeLinker: node-modules)
3. If neither exists, check for `node_modules` directory existence itself — if `node_modules` exists, compare its mtime; if it does not exist, return true (stale)
4. If a sentinel file IS found, compare its mtime against max(package.json, yarn.lock) mtime as before
5. If node_modules exists but no sentinel found, use the `node_modules/.cache` directory mtime as fallback, or just treat as not-stale (yarn install was run but with an unknown yarn version)

Actually, the simplest robust approach: check if `node_modules` directory exists at all. If it doesn't, stale. If it does, check for sentinel files (.yarn-integrity OR .yarn-state). If a sentinel exists, compare its mtime against lock files. If no sentinel exists but node_modules exists, consider it NOT stale (benefit of the doubt — we can't reliably detect staleness without a sentinel).

This way:
- Fresh clone with no node_modules → stale (correct)
- After yarn install (v1) → .yarn-integrity exists, mtime comparison works
- After yarn install (berry) → .yarn-state exists, mtime comparison works
- After yarn install (unknown) → node_modules exists, no sentinel → not stale (safe default)
- After editing package.json → sentinel mtime < package.json mtime → stale (correct)

Add a debug log line (tracing::debug) when check_stale returns true so future debugging is easier.
