## Summary

Phase 05.1 is a large UX overhaul on the existing Rust/Ratatui codebase. The work is entirely within the established architecture: no new external crates are required. Every feature maps to extending well-understood patterns: adding to `Action` enum, updating `AppState` fields, extending `handle_key()` routing, refactoring `update()`, and modifying UI render functions.

The most structurally significant changes are: (1) the command queue replacing `pending_restart: bool` and `pending_command_after_install: Option<CommandSpec>` with a proper `VecDeque<CommandSpec>`, (2) the layout restructure moving the worktree list from the left column to a bottom row and replacing the `List` widget with `Table`, (3) the multiplexer abstraction extracting `src/infra/tmux.rs` into a trait with two implementations, and (4) output persistence keyed by `WorktreeId` replacing the single shared `command_output` deque.

The complete keybinding rework replaces two flat palette modes (`PaletteMode::Git`, `PaletteMode::Rn`) with five submenu modes (`Android`, `Ios`, `Clean`, `Sync`, `Git`), each following the same two-stroke pattern already established. The clean submenu adds a novel toggle-state pattern (check/uncheck items before confirming) which needs a new `CleanOptions` struct in `AppState`. The sync-before-run prompt replaces the silent lazy-install behavior of WORK-06 with an explicit user prompt that feeds into the command queue.

**Primary recommendation:** Break this phase into 7-8 plans along feature boundaries. Work from the inside out вҖ” data structures first (queue, persistence, multiplexer trait), then wiring (handle_key, update), then UI (table layout, footer, modals, help).

### Pattern 1: Command Queue

**What:** Replace `pending_restart: bool` + `pending_command_after_install: Option<CommandSpec>` with a proper FIFO queue.

```rust
// In Action enum вҖ” add:

// CommandExited already exists вҖ” on receipt, pop next item from queue and dispatch

// In update() вҖ” CommandExited arm:
Action::CommandExited => {

    state.command_task = None;

    if let Some(next) = state.command_queue.pop_front() {
        dispatch_command(state, next, metro_tx);
    }
}
```

### Pattern 3: Multiplexer Trait

**What:** Extract `src/infra/tmux.rs` into a trait with TmuxAdapter and ZellijAdapter.

**Implementation:**

```rust
// src/infra/multiplexer.rs
pub trait Multiplexer: Send + Sync + std::fmt::Debug {
    fn new_window(&self, path: &std::path::Path, name: &str, command: &str) -> anyhow::Result<()>;
    fn is_available(&self) -> bool;
}

#[derive(Debug)]
pub struct TmuxAdapter;
impl Multiplexer for TmuxAdapter {
    fn new_window(&self, path, name, command) -> anyhow::Result<()> {
        // existing open_claude_in_worktree() logic, generalized
        std::process::Command::new("tmux")
            .args(["new-window", "-d", "-c", path_str, "-n", name, command])
            .status()?; ...
    }
    fn is_available(&self) -> bool { std::env::var("TMUX").is_ok() }
}

#[derive(Debug)]
pub struct ZellijAdapter;
impl Multiplexer for ZellijAdapter {
    fn new_window(&self, path, name, command) -> anyhow::Result<()> {
        // zellij action new-tab --name <name> --cwd <path> [then run command]
        // Zellij tab creation: `zellij action new-tab --name <name>`
        // then run command in it вҖ” exact CLI varies, needs validation
        ...
    }
    fn is_available(&self) -> bool { std::env::var("ZELLIJ").is_ok() }
}

// Auto-detect function:
pub fn detect_multiplexer() -> Option<Box<dyn Multiplexer>> {
    if std::env::var("TMUX").is_ok() { return Some(Box::new(TmuxAdapter)); }
    if std::env::var("ZELLIJ").is_ok() { return Some(Box::new(ZellijAdapter)); }
    None
}

// In AppState:
// pub multiplexer: Option<Box<dyn Multiplexer>>  // replaces tmux_available: bool
```

**Note on Zellij CLI:** Zellij's `new-tab` command needs `--cwd` and ability to run a command. The exact invocation is `zellij action new-tab --name <name> --cwd <path>` (creates tab in that dir), then optionally run command via shell. Exact flags should be validated at implementation time вҖ” the adapter pattern means this is isolated.

### Pattern 4: Worktree Table with TableState

**What:** Replace `List` + `ListState` rendering in `panels.rs` with `Table` + `TableState`.

**Implementation:**
```rust

// In panels.rs (new render_worktree_table):
use ratatui::widgets::{Table, Row, Cell, TableState};

fn render_worktree_table(f: &mut Frame, area: Rect, state: &mut AppState) {
    let rows: Vec<Row> = state.worktrees.iter().enumerate().map(|(i, wt)| {
        // Pin metro-active worktree to top (sort handled at data prep, not render)
        let label = wt.label.as_deref().unwrap_or("");
        let name = wt.path.file_name()...;
        let branch = truncate(&wt.branch, branch_width);
        let ticket = extract_ticket_num(&wt.branch).unwrap_or("");
        let title = truncate(wt.jira_title.as_deref().unwrap_or(""), title_width);
        let icons = build_icon_cells(wt);
        Row::new([label, name, branch, ticket, title, icons])
    }).collect();

    let table = Table::new(rows, [
        Constraint::Length(12),   // Label
        Constraint::Length(20),   // Worktree name
        Constraint::Min(15),      // Branch (truncates first)
        Constraint::Length(8),    // Ticket #
        Constraint::Min(20),      // Title (truncates second)
        Constraint::Length(4),    // Icons (fixed вҖ” no header)
```

### Anti-Patterns to Avoid

- **Expanding `pending_restart: bool` further:** The boolean chain (`pending_restart`, `pending_switch_path`, `pending_command_after_install`) is being replaced by the command queue. Do not add more booleans вҖ” funnel everything through the queue.
- **Mutating `state.worktrees` order at render time:** Sort once at data-load, not in `render_worktree_table()` which is called every frame.
- **Adding multiplexer logic directly in `update()`:** `update()` must stay sync; multiplexer calls go in `tokio::spawn` blocks, same as `OpenClaudeCode` today.
- **Clearing command output on new command:** CONTEXT.md is explicit вҖ” output persists. Only clear when user explicitly triggers `CommandOutputClear`.
- **Using `String` for WorktreeId keys:** `WorktreeId` is already a newtype вҖ” use it as the HashMap key (it derives `Hash + Eq`).

## Common Pitfalls

### Pitfall 1: Key Collision in New Scheme

**What goes wrong:** The new `x` key opens the Clean submenu from the worktree list, but in the log panel `x` clears the log. In the metro pane, `x` stops metro. Three different behaviors for the same key.

**Why it happens:** Context-sensitive keybindings share a keyspace.

**How to avoid:** The `handle_key()` dispatch order already handles this вҖ” panel-specific blocks run before global blocks. Add a `FocusedPanel::LogPanel` variant (or re-use `MetroPane` with a sub-state flag) so the clean submenu's `x` trigger only fires when `focused_panel == WorktreeList`. Verify the full key table has no remaining collisions before implementing.

**Warning signs:** Test matrix: for each key in the new scheme, verify behavior across all `FocusedPanel` values and all `PaletteMode` values.

### Pitfall 2: Command Queue Race with CommandCancel

**What goes wrong:** User cancels a running command. The queue still holds the remaining items. Next pop from queue starts immediately after cancel, which may not be what the user wanted.

**Why it happens:** `CommandCancel` + `CommandExited` triggers queue pop.

**How to avoid:** `CommandCancel` should also clear the queue (or the queue should be explicitly clearable). Per CONTEXT.md, queue is "clearable (all-or-nothing)". Add a `CommandQueueClear` action dispatched from `CommandCancel`, or check if queue should auto-clear on cancel vs. continue. The CONTEXT.md says cancel is for "currently running command" вҖ” queue continues. But if user cancels, they may want to stop all. Recommend: `CommandCancel` clears queue too (safe default; user can re-queue if needed).

**Warning signs:** If `CommandExited` is received after `CommandCancel`, check if running_command is None before popping queue.

### Pitfall 3: WorktreeId Stability During Table Sort

**What goes wrong:** Metro-active worktree is pinned to top by sorting `state.worktrees`. The `worktree_table_state.selected()` index no longer refers to the same worktree after a sort.

**Why it happens:** `TableState` stores a selected index, not a `WorktreeId`.

**How to avoid:** After every sort (in `WorktreesLoaded` and wherever metro status changes), re-derive the selected index from the selected worktree's `WorktreeId`. Store `selected_worktree_id: Option<WorktreeId>` in `AppState` as the stable selection reference. Derive the table row index from it at render time.

**Warning signs:** After metro starts in a worktree, the selection appears to jump to a different worktree.

### Pitfall 4: Zellij Tab Command vs. Shell Command

**What goes wrong:** Zellij's `new-tab` subcommand doesn't accept an initial command the same way tmux does.

**Why it happens:** Zellij tab creation and command execution are separate concerns in Zellij's CLI.

**How to avoid:** For Zellij, the pattern is likely: `zellij action new-tab --name <name> --cwd <path>`, which creates a tab with a fresh shell. Running an initial command requires either `zellij run` (which doesn't create a persistent tab) or attaching to the tab and sending input. The ZellijAdapter may need to accept that Claude Code cannot be pre-launched вҖ” it creates the tab at the right CWD and the user runs `claude` themselves. Flag this as a known limitation in the adapter's `new_window` doc comment.

**Warning signs:** Zellij docs show different subcommands for tab creation vs. command execution.

### Pitfall 5: Output Persistence HashMap Key Match

**What goes wrong:** `command_output_by_worktree` misses entries because `WorktreeId` doesn't match across refreshes.

**Why it happens:** `WorktreeId` is derived from the full path string. Path strings must be identical across `WorktreesLoaded` callbacks for the same physical worktree.

**How to avoid:** `WorktreeId(path_str.to_string())` in `worktrees.rs` вҖ” the path comes from `git worktree list --porcelain` output which is canonical. Confirmed stable as long as the worktree directory is not moved. This is acceptable.

**Warning signs:** Command output disappears on refresh. Check that `WorktreeId` is derived identically in `parse_worktree_porcelain()` on every call.

### Pitfall 6: TableState vs. ListState Selection API

**What goes wrong:** `TableState` is used where `ListState` was expected, or vice versa.

**Why it happens:** The worktree list currently uses `ListState` (`worktree_list_state: ratatui::widgets::ListState`). Phase 05.1 replaces this with `TableState`.

**How to avoid:** Find all references to `worktree_list_state` in `app.rs` (selection index reads in `WorktreeSelectNext`, `WorktreeSelectPrev`, `CommandRun`, `StartSetLabel`, `WorktreeSwitchToSelected`, `OpenClaudeCode`) and update to `worktree_table_state`. Both use `select(Some(idx))` and `selected()` вҖ” API is identical, just rename.

**Warning signs:** `render_stateful_widget(list, area, &mut state.worktree_list_state)` вҖ” must become `render_stateful_widget(table, area, &mut state.worktree_table_state)`.

### Pitfall 7: `pending_restart` / `pending_command_after_install` Removal

**What goes wrong:** Removing the old ad-hoc booleans before all their callers are migrated to the queue.

**Why it happens:** `pending_restart` is set in `MetroRestart`, `MetroStart` (while running), and checked in `MetroExited`. `pending_command_after_install` is set in `CommandRun` (WORK-06 path) and consumed in `CommandExited`.

**How to avoid:** The queue subsumes both. Restart chain: queue `[MetroStop, MetroStart]` (conceptually вҖ” metro is not a `CommandSpec`, so the restart chain stays as `pending_restart: bool` or is handled differently). Actually: metro restart is NOT a CommandSpec вҖ” it uses the existing bool. The queue replaces only `pending_command_after_install`. `pending_restart` stays for the metro restart chain because metro is separate from the command queue. Only `pending_command_after_install` is fully replaced by the queue.

**Warning signs:** If metro restart breaks after removing `pending_restart`, you removed it too aggressively.

```rust
pub fn check_stale_pods(worktree_path: &Path) -> bool {
    let pods_dir = worktree_path.join("ios").join("Pods");
    let podfile_lock = worktree_path.join("ios").join("Podfile.lock");

    let pods_mtime = match std::fs::metadata(&pods_dir).and_then(|m| m.modified()) {
        Ok(t) => t,
        Err(_) => return true, // Pods/ absent вҶ’ stale
    };

    match std::fs::metadata(&podfile_lock).and_then(|m| m.modified()) {
        Ok(lock_mtime) => pods_mtime < lock_mtime,
        Err(_) => false, // no Podfile.lock вҶ’ can't determine
    }
}
```

## Open Questions

1. **GitFetch as standalone CommandSpec or inline before GitResetHardFetch**
   - What we know: `g>X` should "always fetch first, then reset to origin/same-branch"
   - What's unclear: Should fetch be a separate CommandSpec enqueued before the reset (giving output for both), or should `build_argv()` in command_runner.rs handle the two-step?
   - Recommendation: Add `GitResetHardFetch` as a single `CommandSpec` but implement it in `command_runner.rs` as a two-step sequence (fetch then reset) within the same task. This gives clean "fetch" output followed by "reset" output, without needing the queue for this specific case.

2. **Fullscreen mode panel focus during fullscreen**
   - What we know: `f` toggles fullscreen for "focused pane (log, metro, command output)"
   - What's unclear: Can you fullscreen the worktree table? CONTEXT.md says "log, metro, command output" вҖ” likely not the table.
   - Recommendation: Fullscreen only applies to MetroPane and CommandOutput. LogPane (if visible) also fullscreenable. Planner documents this constraint.

3. **Android release build UX (Claude's discretion)**
   - What we know: `a>r` вҶ’ `gradlew assembleRelease + adb install`
   - What's unclear: Should these be two separate CommandSpecs queued in order, or a single ShellCommand wrapping both?
   - Recommendation: Two CommandSpecs: `GradlewAssembleRelease` and `AdbInstall`, queued in order. This gives per-step output visibility and allows the user to see if the build failed before install.

4. **Zellij new-tab command invocation**
   - What we know: `$ZELLIJ` env var is set when inside a Zellij session
   - What's unclear: Exact CLI for creating a tab with initial command. `zellij action new-tab --name X --cwd Y` exists; running an initial command may require `zellij action write` or shell trickery.
   - Recommendation: Validate at implementation time. Fall back to: create tab at CWD, no auto-launch command. The user can still type `claude` manually.

5. **Footer icon legend vs. key hints space**
   - What we know: Footer is 1 line. Legend needs: `в—Ҹ=metro вҡ =stale`. Key hints for worktree table are numerous in new scheme.
   - What's unclear: Will everything fit on one line on narrow terminals?
   - Recommendation: Legend on far right (right-aligned), key hints on left. Use `Layout::horizontal` with `Constraint::Min(0)` for hints and `Constraint::Length(legend_width)` for legend. Planner can define the exact Constraint splits.
