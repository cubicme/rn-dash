Create `src/domain/metro.rs` with the following types (see 02-RESEARCH.md Pattern 1):

1. `MetroStatus` enum with variants: `Stopped`, `Running { pid: u32, worktree_id: String }`, `Starting`, `Stopping`. Derive Debug, Clone, PartialEq. Implement Default as Stopped.

2. `MetroHandle` struct with fields:
   - `pid: u32`
   - `worktree_id: String`
   - `stdin_tx: tokio::sync::mpsc::UnboundedSender<Vec<u8>>`
   - `stream_task: tokio::task::JoinHandle<()>`
   - `stdin_task: tokio::task::JoinHandle<()>`

   NOTE: MetroHandle lives in domain/ but references tokio types. This is acceptable because MetroHandle is an infrastructure-bridging type — the domain invariant (single instance) is enforced by MetroManager's Option wrapper, not by MetroHandle's contents. Add a comment explaining this architectural choice. Do NOT add tokio as a dependency of domain — tokio is already in Cargo.toml at the workspace level.

3. `MetroManager` struct with fields:
   - `handle: Option<MetroHandle>` (private)
   - `pub status: MetroStatus`

   Methods:
   - `pub fn new() -> Self` — returns Stopped state
   - `pub fn is_running(&self) -> bool` — self.handle.is_some()
   - `pub fn register(&mut self, handle: MetroHandle)` — asserts handle is None, sets handle and status to Running
   - `pub fn clear(&mut self)` — sets handle to None, status to Stopped
   - `pub fn send_stdin(&self, bytes: Vec<u8>) -> anyhow::Result<()>` — sends via stdin_tx if handle exists
   - `pub fn set_starting(&mut self)` — sets status to Starting
   - `pub fn set_stopping(&mut self)` — sets status to Stopping
   - `pub fn take_handle(&mut self) -> Option<MetroHandle>` — takes the handle for kill operations

Update `src/domain/mod.rs` to add `pub mod metro;`.

  - spawn_metro: uses `tokio::process::Command::new("yarn").args(["start", "--reset-cache"])` with `.current_dir()`, `.process_group(0)`, `.kill_on_drop(true)`, `.stdout/stderr/stdin(Stdio::piped())`

Extend `src/action.rs` Action enum with these new variants (see 02-RESEARCH.md "Action Enum Additions"):

```rust
// Metro control (user-triggered)
MetroStart,
MetroStop,
MetroRestart,
MetroToggleLog,       // l key
MetroScrollUp,        // k when log panel focused
MetroScrollDown,      // j when log panel focused
MetroSendDebugger,    // J (shift-j, when metro pane focused)
MetroSendReload,      // R (shift-r, when metro pane focused)

// Metro background events (not user-triggered)
MetroLogLine(String),
MetroExited,
```

The MetroLogLine variant contains a String, so Action can no longer derive PartialEq trivially — either keep the derive (String is PartialEq) or implement manually. String IS PartialEq so the derive still works. Keep `#[derive(Debug, Clone, PartialEq)]`.

Extend `src/app.rs` AppState with these fields (see 02-RESEARCH.md "AppState Extensions"):

```rust
pub struct AppState {
    // ... existing Phase 1 fields ...

    // Metro state
    pub metro: crate::domain::metro::MetroManager,

    // Log panel
    pub metro_logs: std::collections::VecDeque<String>,
    pub log_scroll_offset: usize,
    pub log_panel_visible: bool,
    pub log_filter_active: bool,

    // Active worktree (stub until Phase 3 populates the real list)
    pub active_worktree_path: Option<std::path::PathBuf>,
}
```

Add `const MAX_LOG_LINES: usize = 1000;` at the top of app.rs.

Update the `Default` impl for AppState — remove `#[derive(Default)]` and implement manually since MetroManager has its own `new()`. Set metro_logs to `VecDeque::new()`, all other new fields to their zero/None/false defaults.

Add stub match arms in `update()` for ALL new Action variants. Each arm should have a comment `// Plan 02 implements runtime behavior` and be a no-op for now, EXCEPT:
- `MetroToggleLog` — toggles `state.log_panel_visible`
- `MetroScrollUp` — decrements `state.log_scroll_offset` (saturating_sub(1))
- `MetroScrollDown` — increments `state.log_scroll_offset` (capped at metro_logs.len())
- `MetroLogLine(line)` — pushes to metro_logs VecDeque, pops front if len > MAX_LOG_LINES
- `MetroExited` — calls `state.metro.clear()`

These are pure state mutations that don't need async runtime, so they belong here not in Plan 02.

Do NOT modify `handle_key()` yet — Plan 02 adds the metro-pane keybinding branch with proper context (it needs to know about the async spawn/kill flow).
